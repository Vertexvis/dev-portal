Build your first application

You have now rendered your first scene and imported your data. Next, let's build an application you can deploy publicly!

We'll be using the Vertex Web UI Components, which are standards-based web components. You could create your app using plain JavaScript, but we'll use NextJS, a React-based framework with a great developer experience.

To get an idea of what you'll be building, check out this example application we created.

Before we start#

The following diagram gives an overview of how your applications interact with the Vertex Cloud.

In the Render your first scene guide, we focused on the Your Application section, communicating with the Vertex WebSocket using Vertex's test stream key. In the Import data guide, instead of having you create an API in the Your Cloud section, we used the Vertex CLI, which uses the Vertex API Client to communicate with the Vertex REST API using OAuth tokens.

In this guide, we'll create a deployable application with an API. With this application, you can securely talk to both the Vertex WebSocket and Vertex REST API. Using these pieces, the full power of the Vertex platform is in your hands.

Creating your application#

Use our NextJS Starter template on GitHub. Press the Use this template button to create a new repository as your starting point. If you'd rather use something other than GitHub, simply clone the repository.

Once your repository exists, start the application with Yarn.

# Install dependencies
yarn install
# Start development server
yarn dev

Browse to http://localhost:3000 to see your application displaying the Vertex test scene!

Opening your scene#

You can open different scenes in your application using the Open Scene button. Each scene requires both your Vertex client ID and a stream key.

To create a stream key, you'll need your scene ID from the Import data guide. If you have it handy, great! If not, you can list your scenes with the CLI and copy the ID.

vertex scenes:list --extended

Stream keys grant Vertex's Viewer UI component access to your scene. The default stream key expiration is 10 minutes (600 seconds). You can increase this to a maximum of 24 hours (86000 seconds) using the --expiry flag.

vertex stream-keys:create --sceneId [YOUR_SCENE_ID] --expiry 86000

Now, in your application, click Open Scene, enter your client ID and stream key, and click Open Scene. You'll now see your scene instead of the test scene.

Understanding the code#

Start by opening src/pages/index.tsx. We'll walk through the Home function in pieces.

const router = useRouter();
const viewer = useViewer();
const [credentials, setCredentials] = React.useState<
StreamCredentials | undefined
>();
const [dialogOpen, setDialogOpen] = React.useState(false);
const [properties, setProperties] = React.useState<Properties>({});
// Prefer credentials in URL to enable easy scene sharing. If empty, use defaults.
React.useEffect(() => {
if (!router.isReady) return;
setCredentials({
clientId: head(router.query.clientId) || DefaultCredentials.clientId,
streamKey: head(router.query.streamKey) || DefaultCredentials.streamKey,
});
}, [router.isReady]);
// On credentials changes, update URL.
React.useEffect(() => {
if (credentials) router.push(encodeCreds(credentials));
}, [credentials]);
// Open dialog if 'o' key pressed
const keys = useKeyListener();
React.useEffect(() => {
if (!dialogOpen && keys.o) setDialogOpen(true);
}, [dialogOpen, keys]);

At the top, we configure variables we'll need later. We also use React effect hooks to update the URL on credential changes and listen for 'o' key presses to open the Open Scene dialog.

Now, the fun stuff. We return a JSX element representing your application.

return router.isReady && credentials ? (
<Layout
header={<Header onOpenSceneClick={() => setDialogOpen(true)} />}
main={
viewer.isReady && (
<Viewer
configEnv={Env}
credentials={credentials}
onSelect={async (hit) => {
setProperties(toProperties({ hit }));
await selectByHit({ hit, viewer: viewer.ref.current });
}}
viewer={viewer.ref}
/>
)
}
rightDrawer={<RightDrawer files={files} properties={properties} />}
>
{dialogOpen && (
<OpenDialog
credentials={credentials}
onClose={() => setDialogOpen(false)}
onConfirm={(cs) => {
setCredentials(cs);
setDialogOpen(false);
}}
open={dialogOpen}
/>
)}
</Layout>
) : (
<></>
);

Let's break this down to see exactly what's happening:

  • Once the NextJS router is ready and credentials are set, we return the application's components wrapped in a Layout. Until then, we return an empty React fragment.
  • The header includes the Open Scene button. With the onOpenSceneClick callback, we set the dialogOpen state.
  • Next is the main application content:
    • When the viewer is ready, we display the Viewer component.
    • With the onSelect callback, we set the properties state. Then, call a utility function similar to the Render your first scene guide's function to perform scene item selection.
  • The right drawer receives files, which we'll discuss more later, and metadata properties for display.
  • Finally, the Open Scene dialog. It's shown if the dialogOpen React state is true. We pass two callbacks:
    • onClose closes the dialog.
    • onConfirm sets the credentials state, then closes the dialog.

Calling the Vertex API#

Open the Recent Files accordion in the right drawer. You should see "No data". Similarly, browse to http://localhost:3000/api/files and you should see a 401: Unauthorized error. The application doesn't know your Vertex client ID and secret, so it cannot make API calls on your behalf. Let's fix that.

In your terminal, type Ctrl+C to stop the development server. NextJS has built-in support for environment variables, and the Vertex NextJS Starter comes with a .env.local.template file you can rename. This makes it easier for you to set them.

mv .env.local.template .env.local

Open .env.local and set the VERTEX_CLIENT_ID and VERTEX_CLIENT_SECRET to your Vertex client ID and secret. You can leave NEXT_PUBLIC_VERTEX_ENV blank to use the default. Since the environment variables don't start with NEXT_PUBLIC_, they are only exposed server-side, keeping your client secret safe.

note
Your Vertex client secret is sensitive and must be kept secure at all times. You should never include it in client-side web applications.

Now, start the development server again with yarn dev and browse to http://localhost:3000/api/files. You should see a list of files you've uploaded! You'll see a simplified list in the Recent Files accordion.

How does this work? Go ahead and open src/pages/api/files.ts.

export default async function getFiles(
req: NextApiRequest,
res: NextApiResponse<FileList | Failure>
): Promise<void> {
return makeCall(res, (client) => client.files.getFiles({ pageSize: 5 }));
}

This is the function getting called when you browse to http://localhost:3000/api/files. We're using NextJS API Routes and the makeCall helper function. Open src/lib/vertex-api.ts to see what that does.

export async function makeCall<T>(
res: NextApiResponse<T | Failure>,
apiCall: (client: VertexClient) => Promise<AxiosResponse<T>>
): Promise<void> {
try {
const c = await getClient();
res.status(200).json((await apiCall(c)).data);
} catch (error) {
res.status(500).json(
error.vertexError?.res ?? {
errors: [{ status: '500', title: 'Unknown error from Vertex API.' }],
}
);
}
}
let Client: VertexClient | undefined;
async function getClient(): Promise<VertexClient> {
if (Client != null) return Client;
Client = await VertexClient.build({
client: {
id: process.env.VERTEX_CLIENT_ID ?? '',
secret: process.env.VERTEX_CLIENT_SECRET ?? '',
},
});
return Client;
}

makeCall calls getClient, which creates a Vertex API Client if it doesn't already exist using the environment variables we provided. It then uses the client to call the Vertex API and handles possible errors.

Open src/pages/index.tsx once more. At the bottom, we're using NextJS's getServerSideProps function for server-side rendering.

export const getServerSideProps: GetServerSideProps = async (context) => {
const noFiles = { props: { files: [] } };
const host = context.req.headers.host;
if (!host) return noFiles;
const baseUrl = `http${host.startsWith('localhost') ? '' : 's'}://${host}`;
const res = await (await fetch(`${baseUrl}/api/files`)).json();
return res == null || res.errors
? noFiles
: { props: { files: toFileData(res) } };
};

Since fetch requires absolute URLs, we get the host from the request and construct a baseUrl with either http:// or https:// depending on if we're running locally or not. Then, we make a request to our /files API. We map the response with toFileData and the NextJS passes the result as React props into our Home component. As we saw above, we pass the files prop to the right drawer for display.

Deploying your application#

Follow one of these guides to deploy your application. If you're not sure which to choose, we recommend Vercel.

  • Vercel: From the creators of NextJS, they know best how to deploy their framework. It's as easy as linking your GitHub/GitLab/Bitbucket account, adding environment variables, and visiting your site! On each git push, Vercel automatically deploys a new version using Vercel Serverless Functions for APIs.
  • Netlify: Similar to Vercel and just as easy. They use Netlify Functions for APIs.
  • AWS CDK: If you already have sites deployed in AWS, this CDK construct helps deploy NextJS, using Lambda@Edge for APIs.

Wrapping up#

Congratulations! To recap the concepts learned so far:

  • The NextJS framework uses React. It allows us to create client-side and server-side portions of applications, giving you the full power of the Vertex platform.
  • Your application uses Vertex UI Components and makes calls to its own API. The UI Components communicate with the Vertex WebSocket using stream keys. Your API uses the Vertex API Client and OAuth tokens to communicate with the Vertex API.
  • There are many deployment options to deploy and share applications built with Vertex.
